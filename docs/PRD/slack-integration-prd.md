## Product Summary

### Solution Overview

This feature connects each PRD's lifecycle to a dedicated Slack channel, making Slack the ambient collaboration layer for review and refinement without requiring stakeholders to log into the web application to participate. When a PRD is submitted for review (status transitions to `IN_REVIEW`), the system automatically creates a Slack channel named after the PRD, posts the current PRD document as a file in that channel, and begins bidirectional message synchronisation between the PRD Web Agent's Chat tab and the Slack channel. When a PRD re-enters the refinement phase, the accumulated Slack conversation history is included as context for the AI agent, ensuring the agent is aware of all reviewer feedback regardless of whether it was entered in the app or in Slack.

Slack workspace connection is managed via a per-organisation OAuth app installation flow. A project admin installs the PRD Web Agent Slack app once and all PRDs in that workspace benefit from the integration.

### Value Proposition

- **Zero-barrier reviewer participation** — Approvers and stakeholders can leave feedback in Slack without learning a new tool, dramatically lowering the review participation rate
- **Persistent, contextual conversation** — All PRD discussion lives in one place — a Slack channel that is named after and tied to the PRD — eliminating scattered DMs and email threads
- **Richer AI refinements** — The AI agent ingests Slack conversation history at refinement time, so reviewer comments left in Slack directly improve the next revision without the PM needing to manually relay them
- **Automatic coordination overhead reduction** — Channel creation, document posting, and message synchronisation happen automatically at status transition; PMs and reviewers spend zero time on setup

## Problem Statement

Today, PRD review in the PRD Web Agent is contained entirely within the web application. Reviewers must log in, navigate to the PRD, and post comments or Chat messages there. In practice, organisations that already use Slack default to discussing PRDs there — outside the tool — because the friction of switching contexts is too high. This creates two failure modes:

1. **Fragmented feedback** — Review conversations occur in Slack DMs, shared channels, or email rather than in the PRD tool, so the PM must manually reconcile external feedback with the document
2. **AI context blindness** — The AI agent cannot access Slack-resident feedback during refinement, so refinements generated by the agent do not reflect the real reviewer conversation. PMs must paraphrase reviewer comments into the Chat interface, adding manual overhead and introducing information loss

These failures slow down the review cycle, reduce the quality of AI-assisted refinements, and frustrate both PMs and reviewers who want to use their preferred tools.

## User Analysis

### Target Users

**Primary:** Product Managers managing PRDs through the review and refinement lifecycle in organisations that use Slack
**Secondary:** PRD reviewers and approvers who prefer to engage via Slack rather than the web application

### Persona 1: The Slack-Native PM (Primary)

| Attribute | Detail |
|-----------|--------|
| **Profile** | PM at a mid-size software company where all cross-functional communication happens in Slack |
| **Context** | Submits PRDs for review, expects reviewers to comment, then refines the document with the AI agent |
| **Pain point** | Reviewers ignore the PRD Web Agent's comment system but respond actively in Slack. The PM must copy-paste Slack feedback into the Chat tab to make the agent aware of it before refinement |
| **What changes** | A channel is auto-created when the PRD goes to review; reviewers comment in Slack; the Chat tab mirrors the Slack conversation; the agent reads all of it at refinement time without any PM intervention |

### Persona 2: The Async Reviewer (Secondary)

| Attribute | Detail |
|-----------|--------|
| **Profile** | Engineering lead or design director who is tagged in a Slack channel to review a PRD |
| **Context** | Receives a Slack notification in `#prd-payment-flow-redesign`, reads the posted document, and leaves comments directly in the channel thread |
| **Pain point** | Being asked to log into a separate web application to leave feedback adds friction and delays their response |
| **What changes** | The reviewer's entire interaction is in Slack — they read the document, leave comments, see responses, and never need to visit the web application |

### Persona 3: The Approval-Track PM (Primary)

| Attribute | Detail |
|-----------|--------|
| **Profile** | Senior PM who uses the PRD Web Agent for AI-assisted refinement but oversees a process where approvers are not power users of the tool |
| **Context** | The PRD enters a second refinement round after rejection; the approver left two pages of notes in Slack |
| **Pain point** | The agent's refinement suggestions are generic because it lacks access to the specific reviewer objections that live in Slack |
| **What changes** | When the PM clicks "Refine", the agent automatically receives the Slack channel history as context, producing targeted revisions that address the exact reviewer concerns |

### User Journey (Primary — PM Submitting a PRD for Review)

```
1. PM finishes drafting a PRD in the PRD Web Agent
2. PM clicks "Submit for Review" (status: DRAFT → IN_REVIEW)
3. System automatically:
   a. Creates a Slack channel named #prd-<slugified-title> in the connected workspace
   b. Posts the current PRD content as a .md file in the channel Files section
   c. Posts an introductory message: "PRD '[Title]' has been submitted for review.
      Leave your feedback in this channel — it will be synced back to the PRD tool."
   d. Invites configured reviewer users to the channel (if Slack user IDs are mapped)
4. Reviewers are notified in Slack; they open the channel, read the posted document,
   and leave feedback as Slack messages in the channel
5. The PRD Web Agent's Chat tab (now labelled "Slack") shows the same messages in
   near real-time; the PM can read and reply directly from the web app
6. PM or reviewer approves or rejects the PRD via the web application or a Slack slash command
7. If the PRD is rejected (IN_REVIEW → DRAFT), the PM clicks "Refine" in the web app:
   a. The system fetches the full Slack channel message history
   b. The history is injected as context into the AI refinement prompt
   c. The agent produces targeted revisions that address the reviewer comments
8. PM accepts the refined PRD and resubmits; the existing Slack channel receives a new
   file post of the updated document and a message: "PRD '[Title]' v2 has been
   resubmitted for review."
9. If the PRD is approved or permanently rejected/deleted, the Slack channel is archived
   (not deleted) so the conversation remains searchable
```

## Goals & Non-Goals

### Goals

| # | Goal | Success Metric |
|---|------|---------------|
| G1 | A Slack channel is automatically created when a PRD transitions to `IN_REVIEW` | Channel exists in the connected workspace within 5 seconds of the status transition |
| G2 | The current PRD document is posted to the channel's Files section at submission time | File appears in the Slack channel Files tab; content matches the PRD's current version |
| G3 | The Chat tab on the PRD detail page shows the live Slack channel conversation bidirectionally | Messages sent in the app appear in Slack within 5 seconds; messages sent in Slack appear in the app within 30 seconds |
| G4 | Slack conversation history is injected into the AI agent's context at refinement time | The agent's refinement prompt contains a "Slack Conversation History" section; generated output demonstrably references reviewer comments from Slack |
| G5 | Slack workspace connection is managed via OAuth app installation | A project admin can connect a workspace via a standard OAuth flow without writing code or configuring webhooks manually |
| G6 | Slack channels are archived (not deleted) when a PRD is approved, rejected, or deleted | Channel is in archived state in Slack within 60 seconds of the lifecycle event |
| G7 | Channel naming is deterministic and collision-safe | Channel names follow the `prd-<slug>` pattern; collisions are resolved by appending a short suffix |

### Non-Goals

| # | Non-Goal | Rationale |
|---|----------|-----------|
| N1 | Slack slash commands for approving or rejecting a PRD | Status transitions carry authorization and audit requirements that are simpler to enforce in the web app for v1 |
| N2 | Support for Slack Enterprise Grid (multi-workspace organisations) | Standard single-workspace OAuth is sufficient for the target customer segment in v1 |
| N3 | Syncing file attachments or images posted in Slack into the PRD tool | Text-only message sync is sufficient for v1; binary attachments add complexity and storage cost |
| N4 | Per-message attribution mapping (showing Slack user avatars in the web app) | Display name attribution is sufficient; full identity federation is deferred |
| N5 | Thread-level message organisation in the web app | All messages are shown in a flat chronological feed in the Chat tab; threading is deferred |
| N6 | Manual Slack channel creation for PRDs that remain in DRAFT | Channels are only created on `IN_REVIEW` transition; draft-phase channels are out of scope |
| N7 | Slack DM notifications to individual reviewers | Channel-level notifications via Slack's built-in `@here` or invite mechanism are sufficient for v1 |
| N8 | Retroactive channel creation for PRDs already in `IN_REVIEW` before this feature ships | The integration applies to PRDs that transition to `IN_REVIEW` after the feature is deployed |

## Functional Requirements

### Delivery Approach

All functional requirement groups (FR-1 through FR-7) are treated as a single coherent feature set and ship together in Phase 1 (MVP). The Slack OAuth connection (FR-1), channel lifecycle management (FR-2), document posting (FR-3), bidirectional message sync (FR-4), refinement context injection (FR-5), and authorization/security controls (FR-6, FR-7) must all be present and passing acceptance criteria before the feature is released.

---

### FR-1: Slack Workspace Connection (OAuth App Installation)

- **FR-1.1:** A project admin can connect a Slack workspace from the project settings page (`/projects/[id]/settings`) via a "Connect Slack" button
- **FR-1.2:** Clicking "Connect Slack" initiates a standard Slack OAuth 2.0 app installation flow using the `bot` token scope with the following OAuth scopes: `channels:manage`, `channels:history`, `channels:read`, `chat:write`, `files:write`, `users:read`, `users:read.email`
- **FR-1.3:** On successful OAuth callback, the system stores the workspace's `access_token`, `team_id`, `team_name`, and `bot_user_id` in a new `SlackInstallation` database record associated with the project
- **FR-1.4:** Only one Slack workspace can be connected per project at a time; attempting to connect a second workspace replaces the existing installation after a confirmation prompt
- **FR-1.5:** The project settings page displays the connected workspace name and a "Disconnect" button when an installation exists
- **FR-1.6:** Clicking "Disconnect" revokes the stored token by calling `auth.revoke` on the Slack API and deletes the `SlackInstallation` record; Slack channel synchronisation ceases for all PRDs in the project
- **FR-1.7:** OAuth state parameter is used to prevent CSRF; the state value is a signed, time-limited token tied to the initiating user's session
- **FR-1.8:** The Slack app's redirect URI is configured to `POST /api/slack/oauth/callback`; this route exchanges the OAuth code for a token and stores the installation

---

### FR-2: Slack Channel Lifecycle Management

- **FR-2.1:** When a PRD transitions to `IN_REVIEW` in a project that has a connected Slack workspace, the `StatusWorkflowService` triggers an asynchronous Slack channel creation job
- **FR-2.2:** Channel naming convention: `prd-<slug>` where `<slug>` is the PRD title lowercased, non-alphanumeric characters replaced by hyphens, consecutive hyphens collapsed, and the result truncated to 75 characters (Slack's channel name limit is 80; the `prd-` prefix consumes 4)
- **FR-2.3:** Collision handling: before creating the channel, the system checks whether a channel with the computed name already exists in the workspace. If a collision is detected, a 4-character random alphanumeric suffix is appended (e.g., `prd-payment-flow-a3f2`) and the check is repeated. A maximum of 5 suffix-generation attempts are made; if all collide, the job fails with a structured error log and the PRD status transition still succeeds (Slack integration is non-blocking)
- **FR-2.4:** The channel `id` and `name` are stored in a new `SlackChannel` database record associated with the PRD (`prdId`, `channelId`, `channelName`, `workspaceTeamId`, `createdAt`)
- **FR-2.5:** After channel creation, the system posts an introductory message to the channel using `chat.postMessage`:

  ```
  PRD "*<PRD title>*" has been submitted for review.
  View it in the PRD Web Agent: <app URL>/prd/<id>

  Leave your feedback as messages in this channel — they will appear in the
  PRD tool's Chat tab and will be included as context when the PRD is refined.
  ```

- **FR-2.6:** When a PRD transitions to `APPROVED`, `SUBMITTED`, or is soft-deleted (`isDeleted = true`), the associated Slack channel is archived via `conversations.archive`. Archiving is fire-and-forget; failure is logged but does not block the status transition
- **FR-2.7:** When a PRD transitions back to `IN_REVIEW` from `DRAFT` (a re-submission after rejection), the system checks whether an existing `SlackChannel` record exists for the PRD. If it does, the channel is un-archived via `conversations.unarchive` and a new version notice is posted rather than creating a duplicate channel
- **FR-2.8:** If the Slack workspace is disconnected while a PRD is in `IN_REVIEW`, the `SlackChannel` record is retained in the database but the sync job for that channel is disabled; no further messages are sent or received via Slack for that channel

---

### FR-3: PRD Document Posting to Slack

- **FR-3.1:** Immediately after the introductory message (FR-2.5), the system uploads the PRD's current content to the Slack channel's Files section using `files.uploadV2`
- **FR-3.2:** The file is uploaded as a Markdown file with filename `<prd-slug>-v<version>.md` and the file content is the raw Markdown text of the current `PrdVersion`
- **FR-3.3:** When a PRD re-enters `IN_REVIEW` after revision (FR-2.7), a new file with an incremented version suffix is uploaded to the same channel so reviewers can see the diff in the channel's Files history
- **FR-3.4:** The Slack `file_id` of each uploaded document is stored in the `SlackChannel` record (as a JSON array `postedFileIds`) to enable future retrieval
- **FR-3.5:** If file upload fails, the error is logged and a fallback text message is posted to the channel:

  ```
  Could not upload the PRD document file. View the full document here:
  <app URL>/prd/<id>
  ```

### FR-4: Bidirectional Message Synchronisation

#### FR-4.1: Slack → App (Inbound)

- **FR-4.1.1:** Inbound Slack messages are delivered to the application via the Slack Events API (`event_callback` for `message.channels`). The app subscribes to the `message.channels` event type for the bot's installed workspace
- **FR-4.1.2:** The system exposes a webhook endpoint at `POST /api/slack/events` to receive Slack event payloads. The endpoint verifies each request's `X-Slack-Signature` header using HMAC-SHA256 before processing
- **FR-4.1.3:** On receiving a `message.channels` event for a channel matching a `SlackChannel` record, the system creates a `SlackMessage` database record (`id`, `prdId`, `channelId`, `slackTs` (Slack timestamp), `slackUserId`, `slackUserName`, `text`, `direction: "INBOUND"`, `createdAt`) and emits a Socket.io event to the PRD's room so connected Chat tab clients receive the message in real time
- **FR-4.1.4:** Messages from the bot user itself (`bot_id` matches the installation's `bot_user_id`) are ignored to prevent echo loops
- **FR-4.1.5:** The events endpoint responds with HTTP 200 within 3 seconds to satisfy Slack's acknowledgement requirement. Heavy processing (DB writes, Socket.io emit) happens asynchronously after the response
- **FR-4.1.6:** Slack URL verification challenge (`url_verification` event type) is handled by the endpoint, returning the `challenge` value without authentication

#### FR-4.2: App → Slack (Outbound)

- **FR-4.2.1:** When a user sends a message in the Chat tab while the PRD is in `IN_REVIEW` and the PRD has an associated `SlackChannel` record, the message is forwarded to Slack via `chat.postMessage` in addition to being stored locally
- **FR-4.2.2:** The forwarded message includes the sender's display name as the message username (using Slack's `username` parameter on `chat.postMessage`)
- **FR-4.2.3:** The Chat tab's existing ChatInterface component continues to function for PRDs without a connected Slack channel or not in the `IN_REVIEW` status; the Slack sync layer is additive and non-breaking
- **FR-4.2.4:** Outbound message failures (Slack API error, rate limit) are logged and do not cause the in-app message to fail; the message is stored locally regardless

#### FR-4.3: Message Display in Chat Tab

- **FR-4.3.1:** When the PRD has a connected Slack channel, the Chat tab heading changes to "Slack" to communicate to the user that the conversation is mirrored
- **FR-4.3.2:** Messages are displayed in chronological order, with inbound Slack messages labelled with the Slack user's display name and a Slack icon indicator
- **FR-4.3.3:** On initial load of the Chat tab, the system fetches the last 200 messages from the `SlackMessage` table for the PRD and displays them before the live Socket.io stream begins
- **FR-4.3.4:** If the Slack integration is not configured for the project, the Chat tab displays the existing agent chat interface unchanged

---

### FR-5: Refinement Context Injection

- **FR-5.1:** When a PM clicks "Refine" on a PRD that has a `SlackChannel` record, the refinement prompt builder fetches all `SlackMessage` records for the PRD ordered by `createdAt` ascending
- **FR-5.2:** A maximum of 200 most-recent messages are fetched; if the conversation exceeds this limit, the oldest messages are omitted and a note is prepended to the context block: "Note: showing the 200 most recent messages of a longer conversation."
- **FR-5.3:** The fetched messages are formatted and appended to the agent's refinement prompt as a clearly labelled context block:

  ```
  ## Slack Reviewer Conversation

  The following messages are from the Slack channel #<channel-name> where
  reviewers discussed this PRD. Use this conversation to understand reviewer
  concerns, requested changes, and areas of agreement. Address specific
  objections and suggestions where relevant.

  [<timestamp>] <display-name>: <message text>
  [<timestamp>] <display-name>: <message text>
  ...
  ```

- **FR-5.4:** If the PRD has no `SlackChannel` record or the `SlackMessage` table contains no messages for the PRD, no Slack context block is appended; the refinement prompt is unaffected
- **FR-5.5:** The total text size of the Slack context block is capped at 40,000 characters. If the formatted conversation exceeds this limit, messages are truncated from the oldest end until the limit is satisfied
- **FR-5.6:** The `buildRefinePrompt()` function (or equivalent refinement prompt builder) is extended to accept an optional `slackMessages` parameter following the same pattern as the existing `attachments` parameter in `buildCreatePrompt()`

---

### FR-6: Authorization and Access Control

- **FR-6.1:** Only project admins (role `ADMIN`) can connect or disconnect a Slack workspace from the project settings page
- **FR-6.2:** `POST /api/slack/oauth/callback` validates that the state token is signed and has not expired (maximum age: 10 minutes); requests with invalid or expired state tokens return HTTP 400
- **FR-6.3:** `POST /api/slack/events` does not require application authentication but must validate the `X-Slack-Signature` header on every request (FR-4.1.2). Requests with invalid signatures return HTTP 403
- **FR-6.4:** The `SlackInstallation.access_token` is encrypted at rest using AES-256 (the application's `ENCRYPTION_KEY` environment variable); the plaintext token is never written to logs
- **FR-6.5:** Only users who are project members can send Chat tab messages that are forwarded to Slack
- **FR-6.6:** The Slack event webhook endpoint rate-limits inbound event processing per channel to prevent a burst of Slack messages from overloading the database; events exceeding 50 per minute per channel are queued via Redis rather than processed synchronously

---

### FR-7: Settings and Configuration

- **FR-7.1:** A global settings toggle ("Enable Slack integration") is added to the admin settings page (`/settings`); when disabled globally, no Slack channels are created and no messages are synced, regardless of per-project installations
- **FR-7.2:** The project settings page displays the Slack integration status (connected / not connected) with the connected workspace name and team ID
- **FR-7.3:** A per-project toggle ("Auto-invite reviewers to Slack channel") controls whether users with `REVIEWER` or `APPROVER` roles who have a Slack user ID mapped in the system are automatically invited to newly created channels. The default is off
- **FR-7.4:** A `SlackUserMapping` table stores the association between a PRD Web Agent `userId` and a Slack `userId` (`slackUserId`); mappings are created when a user authenticates with OAuth and their email matches a Slack workspace member, or manually by an admin
- **FR-7.5:** Environment variables required for the integration: `SLACK_CLIENT_ID`, `SLACK_CLIENT_SECRET`, `SLACK_SIGNING_SECRET`, `SLACK_APP_URL` (public URL for OAuth redirect and message links)

| NFR-12 | **Backward compatibility** | PRDs without a Slack integration, and projects without a connected workspace, behave identically to the current implementation | No changes to existing Chat tab or status transition APIs for non-Slack-enabled projects |
| NFR-13 | **Graceful degradation** | If the Slack API is unavailable, all PRD Web Agent functionality continues normally; Slack operations are skipped with a warning log | Integration tests simulate Slack API unavailability; status transitions and refinements succeed |

## Acceptance Criteria

### AC-1: Slack Workspace Connection

- [ ] **AC-1.1:** A project admin navigates to project settings and clicks "Connect Slack"; they are redirected to Slack's OAuth authorization page with the correct scopes and a valid state parameter
- [ ] **AC-1.2:** After authorising in Slack, the admin is redirected back to the project settings page and a success banner reads "Slack workspace '[workspace name]' connected successfully"
- [ ] **AC-1.3:** The project settings page shows the connected workspace name and a "Disconnect" button
- [ ] **AC-1.4:** A non-admin project member cannot see or access the Slack connection controls; the UI hides the section and direct API calls to the OAuth flow return HTTP 403
- [ ] **AC-1.5:** Clicking "Disconnect" shows a confirmation prompt; on confirmation, the workspace is disconnected, the settings page returns to "Connect Slack" state, and no further Slack operations occur for PRDs in that project
- [ ] **AC-1.6:** A request to `POST /api/slack/oauth/callback` with an expired or invalid state token returns HTTP 400 and does not store any token

### AC-2: Channel Auto-Creation on Submission

- [ ] **AC-2.1:** A PM submits a PRD titled "Payment Flow Redesign" for review; within 5 seconds a channel named `#prd-payment-flow-redesign` exists in the connected Slack workspace
- [ ] **AC-2.2:** The new channel contains an introductory message linking back to the PRD in the web app and a `.md` file in the Files section containing the PRD's current content
- [ ] **AC-2.3:** A PRD with a very long title (200 characters) produces a channel name of at most 79 characters total (including the `prd-` prefix)
- [ ] **AC-2.4:** Two PRDs with titles that produce the same slug result in channels named `#prd-my-feature` and `#prd-my-feature-<suffix>` respectively; no error occurs and both PRDs function independently
- [ ] **AC-2.5:** When Slack is not configured for the project, submitting a PRD for review succeeds normally; no Slack operations are attempted and no error is shown to the user
- [ ] **AC-2.6:** When Slack channel creation fails (simulated by a mock Slack API error), the PRD status transition still succeeds; a structured error log entry is written with the failure details and the user sees no Slack-related error

### AC-3: PRD Document File Posting

- [ ] **AC-3.1:** The file posted to the Slack channel contains the full Markdown content of the PRD's current version
- [ ] **AC-3.2:** The file is named `<prd-slug>-v1.md` for the first submission
- [ ] **AC-3.3:** When a revised PRD is resubmitted (after rejection), a new file named `<prd-slug>-v2.md` is posted to the same channel; both versions remain in the channel's Files section
- [ ] **AC-3.4:** If `files.uploadV2` fails, a fallback text message containing the PRD URL is posted to the channel instead; the PRD detail page in the web app is unaffected

### AC-4: Inbound Message Sync (Slack → App)

- [ ] **AC-4.1:** A reviewer sends a message in the Slack channel; within 30 seconds the message appears in the PRD's Chat tab with the reviewer's Slack display name
- [ ] **AC-4.2:** The `SlackMessage` record is persisted in the database with correct `prdId`, `channelId`, `slackTs`, `slackUserId`, `slackUserName`, `text`, and `direction: "INBOUND"`
- [ ] **AC-4.3:** A message sent by the app's bot user in Slack does not appear as an inbound message in the Chat tab (no echo loop)
- [ ] **AC-4.4:** A message in an unrelated Slack channel does not appear in any PRD's Chat tab
- [ ] **AC-4.5:** `POST /api/slack/events` with an invalid `X-Slack-Signature` returns HTTP 403 and the message is not stored
- [ ] **AC-4.6:** `POST /api/slack/events` with a `url_verification` event type returns HTTP 200 with the `challenge` value and does not require signature validation

### AC-5: Outbound Message Sync (App → Slack)

- [ ] **AC-5.1:** A PM types a message in the PRD's Chat tab while the PRD is in `IN_REVIEW`; the message appears in the Slack channel within 5 seconds, attributed to the PM's display name
- [ ] **AC-5.2:** The same message is stored in the `SlackMessage` table with `direction: "OUTBOUND"`
- [ ] **AC-5.3:** A Chat tab message sent when the PRD is in `DRAFT` (no active Slack channel) is stored locally only; no Slack API call is made
- [ ] **AC-5.4:** If `chat.postMessage` returns a rate-limit error, the message is queued in Redis and sent after the retry-after delay; the user sees their message in the Chat tab immediately regardless
- [ ] **AC-5.5:** A non-project-member cannot send messages via the Chat API; the request returns HTTP 403 and no Slack message is sent

### AC-6: Chat Tab Display

- [ ] **AC-6.1:** When the PRD has an active Slack channel, the Chat tab heading reads "Slack" instead of "Chat"
- [ ] **AC-6.2:** Opening the Chat tab for a PRD with an active Slack channel displays the last 200 Slack messages in chronological order before any new Socket.io events arrive
- [ ] **AC-6.3:** Inbound Slack messages are visually distinguished from outbound messages (e.g., different background colour, Slack icon indicator, and the sender's display name)
- [ ] **AC-6.4:** When the PRD has no Slack channel (project not connected, or PRD not in review), the Chat tab displays the existing agent chat interface unchanged; no Slack UI elements are visible

### AC-7: Slack Channel Archival

- [ ] **AC-7.1:** A PRD is approved (`IN_REVIEW` → `APPROVED`); within 60 seconds the associated Slack channel is archived (visible as archived in the Slack workspace)
- [ ] **AC-7.2:** A PRD is soft-deleted; the associated Slack channel is archived
- [ ] **AC-7.3:** A PRD is rejected (`IN_REVIEW` → `DRAFT`) and then resubmitted (`DRAFT` → `IN_REVIEW`); the same Slack channel is un-archived and a new version notice is posted; no duplicate channel is created
- [ ] **AC-7.4:** If channel archiving fails (simulated by a mock Slack API error), the PRD status transition still completes and the error is logged without surfacing to the user

### AC-8: Refinement Context Injection

- [ ] **AC-8.1:** A PRD enters refinement after reviewer messages exist in the `SlackMessage` table; inspecting the agent session logs confirms the refinement prompt contains a "## Slack Reviewer Conversation" section with correctly formatted messages
- [ ] **AC-8.2:** The generated refinement content demonstrably addresses a specific point raised in the Slack conversation (e.g., if a reviewer asked "Can you add more detail to the security section?", the refined PRD's security section is expanded)
- [ ] **AC-8.3:** A PRD with no Slack channel or no stored messages enters refinement; the prompt contains no "## Slack Reviewer Conversation" section and the refinement proceeds normally
- [ ] **AC-8.4:** A PRD with 500 Slack messages enters refinement; only the 200 most recent messages are included in the prompt; the injected block does not exceed 40,000 characters; generation completes without error
- [ ] **AC-8.5:** The refinement prompt's Slack context block correctly lists messages in chronological order with timestamps and display names

### AC-9: Authorization

- [ ] **AC-9.1:** An unauthenticated request to `GET /api/projects/[id]/slack` returns HTTP 401
- [ ] **AC-9.2:** A project member (non-admin) calls the Slack disconnect API; the request returns HTTP 403 and the workspace remains connected
- [ ] **AC-9.3:** A project admin calls the Slack disconnect API; the workspace is disconnected and the `SlackInstallation` record is deleted

### AC-10: Observability

- [ ] **AC-10.1:** A successful channel creation produces a structured log entry containing `prdId`, `channelId`, `channelName`, `slackTeamId`, `operation: "channel_create"`, and `durationMs`
- [ ] **AC-10.2:** A failed Slack API call produces a structured log entry at `ERROR` level containing `operation`, `slackErrorCode`, `prdId`, and `channelId`
- [ ] **AC-10.3:** An inbound Slack message produces a log entry at `DEBUG` level containing `slackTs`, `channelId`, `prdId`, and `direction: "INBOUND"`

## Open Questions

| # | Question | Impact | Status |
|---|----------|--------|--------|
| OQ-1 | **Slack App Distribution** — Should the Slack app be a publicly distributed app (listed in the Slack App Directory) or a private app installed only by individual organisations? Public distribution requires Slack review and additional compliance considerations. | Affects the OAuth flow, app verification, and the effort required to support multi-tenant installations | Open — decision needed before starting FR-1 implementation. Engineering and legal should assess the compliance requirements for public distribution vs. a private/internal app model |
| OQ-2 | **Events API vs Socket Mode** — The Slack Events API requires a public HTTPS endpoint; Socket Mode does not but is intended for development use. In production, the application must have a public URL. Is there a deployment constraint (e.g., environments behind a firewall) that would require Socket Mode? | Affects `POST /api/slack/events` endpoint design and the need for a public-facing URL | Open — confirm with infrastructure team that the production environment has a publicly reachable HTTPS endpoint before finalising FR-4.1 |
| OQ-3 | **Message Queue Backend** — FR-6.6 and NFR-4 require a Redis-backed queue for rate-limit protection. The existing stack already uses Redis for Socket.io. Should the Slack message queue use the same Redis instance (a separate key namespace) or a dedicated queue like BullMQ on top of Redis? | Affects operational complexity and queue reliability guarantees | Open — engineering decision during TRD phase. Recommend BullMQ on existing Redis as the lowest-overhead option |
| OQ-4 | **Slack User Email Matching** — FR-7.4 describes auto-mapping PRD Web Agent users to Slack users by email when OAuth is completed. This requires the `users:read.email` scope and a lookup of all workspace members. For large workspaces (1,000+ members), the initial sync could be slow and the `users.list` API is paginated and rate-limited. What is an acceptable latency for the initial mapping population? | Affects the user experience when the workspace is first connected and the complexity of the mapping job | Open — decide whether the mapping should be on-demand (resolve at channel-invite time) or eagerly populated at workspace connection time |
| OQ-5 | **PRD Rejection Channel Behaviour** — When a PRD is rejected (`IN_REVIEW` → `DRAFT`), FR-2.6 specifies archiving the Slack channel. However, FR-2.7 specifies un-archiving it if the PRD is resubmitted. Is it preferable to leave the channel unarchived during `DRAFT` status after rejection, so reviewers can continue discussing, rather than archiving and then un-archiving on resubmission? | Affects the Slack channel lifecycle state machine and reviewer experience between rejection and resubmission | Open — confirm with stakeholders. A possible alternative: archive only on `APPROVED` or hard-deletion, not on rejection |
| OQ-6 | **Slack Conversation History for Refinement — Scope** — FR-5 fetches conversation history from the `SlackMessage` database table (messages that arrived after the integration was connected). If reviewers had a pre-integration Slack conversation about the PRD in another channel, should the PM be able to manually link that channel or paste a conversation export for context? | Affects the scope of FR-5 and the value of the refinement context for early adopters | Open — deferred to Phase 2; manual context injection via the existing document attachment feature is a usable workaround for v1 |
| OQ-7 | **Encryption Key Management** — FR-6.4 requires AES-256 encryption of Slack access tokens at rest. Does the application currently have an `ENCRYPTION_KEY` environment variable and a utility for symmetric encryption, or does this need to be introduced? | Affects both schema design and the set of new utilities required | Open — engineering to confirm whether an encryption utility exists (e.g., from a prior feature) or needs to be built as part of this feature's TRD scope |
